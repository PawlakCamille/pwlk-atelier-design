# Internationalization

Deep reference for the i18n pass in `cami-design-engineer`. Loaded when the diff adds user-facing strings, dates, numbers, or other locale-sensitive output to a codebase that has a translation layer.

For projects with a translation setup (`desktop_fr.json`, `react-i18next`, `next-intl`…), new code must route through it. The findings below.

## Hardcoded User-Facing String

A literal like `'Calendar event'` or `name="Contact"` rendered to the user, bypassing the translation layer. Route it through the `t()` function and translation key the project already uses. The string is invisible in the wrong locale until someone hits that path.

## Date or Number Format Not Locale-Aware

A hardcoded format like `'MMM D, YYYY · h:mm A'`, or a bare `.toLocaleString()` with no locale argument — renders US-style for every user regardless of language. Use the project's localized formatter (`useLocalizedDayjs`, `Intl.*` with the active locale).

## English Leaking Into the A11y Tree

`aria-label`, `alt`, and visually-hidden text are user-facing too. A hardcoded English `aria-label` on an otherwise-localized component stays invisible until a screen-reader user reaches it. Translate them like any other string.
